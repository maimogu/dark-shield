# DarkShield 架构文档

本文档详细介绍 DarkShield 系统的整体架构设计、技术选型和安全性考虑。

## 一、整体架构设计

### 1.1 系统概述

DarkShield 是一个基于 0G 网络的去中心化 DeFi 风险管理平台，采用模块化架构设计，将复杂的风险管理功能分解为多个层次和组件。

### 1.2 架构设计原则

| 设计原则 | 说明 | 实现方式 |
|----------|------|----------|
| 模块化 | 各组件独立可替换 | 合约接口分离、服务解耦 |
| 安全性 | 多重验证保障资产安全 | TEE 验证、Owner 控制、重入锁 |
| 可扩展性 | 支持功能迭代升级 | 代理合约模式、模块化 SDK |
| 透明性 | 操作可追溯验证 | 事件日志、DA 证明存储 |
| 效率性 | 降低 gas 消耗 | 批量操作、事件驱动 |

### 1.3 核心组件职责

#### 前端层（Next.js + TypeScript）

前端采用 Next.js 框架构建，提供用户交互界面。TypeScript 确保类型安全，减少运行时错误。Tailwind CSS 实现响应式设计。

**职责范围**：

- 用户界面渲染和交互处理
- 合约调用封装和状态管理
- 风险数据可视化展示
- 用户钱包连接和签名

#### 计算服务层（0G Compute + Python）

后端服务集成 0G AI 计算能力，提供清算风险预测服务。Flask 框架提供 RESTful API 接口。

**职责范围**：

- 接收和处理预测请求
- 调用 0G AI 模型进行推理
- 生成 DA 证明并存储
- 返回预测结果和证明

#### 智能合约层（Solidity）

智能合约是系统的信任根，部署在 0G Chain 上，提供不可篡改的业务逻辑执行。

**职责范围**：

- 存储和管理预测数据
- 执行风险评估计算
- 处理对冲交易逻辑
- 验证 TEE 证明

## 三、技术选型理由

### 3.1 合约语言：Solidity ^0.8.20

**选择理由**：

1. **生态成熟**：以太坊生态最广泛使用的智能合约语言
2. **工具完善**：Hardhat、Foundry 等开发工具支持完善
3. **安全特性**：内置溢出检查、异常处理等安全机制
4. **升级兼容**：支持代理模式，便于后续功能迭代

**版本选择 0.8.20**：

- 相比 0.8.x 早期版本， gas 优化更好
- 相比 0.9.x，稳定性更好，第三方库兼容性好

### 3.2 开发框架：Hardhat

**选择理由**：

| 特性 | 说明 |
|------|------|
| TypeScript 支持 | 可使用 TypeScript 编写测试和脚本 |
| 插件生态 | 丰富的插件支持（gas 报告、合约验证等） |
| 本地测试网 | 内置 Hardhat Network 支持快速测试 |
| 调试工具 | 强大的堆栈追踪和日志功能 |

### 3.3 依赖库：OpenZeppelin

**使用组件**：

| 组件 | 用途 |
|------|------|
| Ownable | 合约所有权管理 |
| ReentrancyGuard | 重入攻击防护 |
| ERC165 | 接口查询支持 |

**选择理由**：

- 经过广泛审计的安全代码
- 社区认可的标准实现
- 持续维护和更新

### 3.4 前端框架：Next.js + TypeScript

**选择理由**：

| 特性 | 说明 |
|------|------|
| SSR/SSG 支持 | 优化首屏加载和 SEO |
| API Routes | 便于后端 API 集成 |
| 组件化开发 | 代码复用和维护性 |
| TypeScript | 类型安全，减少错误 |

### 3.5 后端框架：Flask

**选择理由**：

- 轻量级，易于部署和维护
- Python 生态丰富，机器学习集成方便
- 0G SDK 支持良好

### 3.6 数据可用性：0G DA

**选择理由**：

- 专为区块链设计的数据可用性层
- 低成本、高吞吐量的数据存储
- 原生支持数据证明和验证

### 3.7 AI 计算：0G Compute

**选择理由**：

- 去中心化的 AI 推理服务
- 可验证的计算结果
- 与 0G DA 原生集成

## 四、安全性考虑

### 4.1 合约安全措施

#### 访问控制

```solidity
// 使用 Ownable 实现管理员控制
contract OgIntegration is Ownable {
    // onlyOwner 修饰符确保只有管理员可以调用敏感函数
    function updatePredictionWithProof(...) external onlyOwner {
        // ...
    }
}
```

**措施说明**：

- 敏感操作仅允许合约所有者调用
- 支持多签名钱包管理 Owner 账户
- 提供紧急暂停机制（可扩展）

#### 重入防护

```solidity
contract RiskHedgeExecutor is ReentrancyGuard {
    // 使用 nonReentrant 修饰符防止重入攻击
    function triggerRiskCheck(address _user) external nonReentrant {
        // ...
    }
}
```

**措施说明**：

- 所有外部调用使用 ReentrancyGuard 保护
- 遵循 Checks-Effects-Interactions 模式
- 状态变更在外部调用之前完成

#### 参数验证

```solidity
function requestPrediction(
    uint256 _healthFactor,
    uint256 _totalDebt,
    uint256 _totalCollateral
) external nonReentrant {
    if (_healthFactor == 0) {
        revert ZeroHealthFactor();
    }
    // ...
}
```

**措施说明**：

- 所有输入参数进行严格验证
- 使用自定义错误类型，便于调试
- 明确的错误消息和错误代码

### 4.2 数据安全

#### DA 证明验证

```solidity
function verifyDAProof(bytes32 _daProof) external view returns (bool) {
    return daProofs[_daProof];
}
```

**措施说明**：

- 所有预测数据关联 DA 证明
- 证明哈希链上存储，可公开验证
- 支持数据溯源和审计

#### 数据完整性

**措施说明**：

- 使用时间戳记录数据生成时间
- 预测结果不可篡改
- 历史记录可追溯查询

### 4.3 操作安全

#### 冷却期机制

```solidity
function _checkCooldown(address _user, uint256 _cooldownPeriod) internal view {
    if (_cooldownPeriod > 0) {
        uint256 lastAction = lastActionTime[_user];
        if (lastAction > 0) {
            uint256 elapsed = block.timestamp - lastAction;
            if (elapsed < _cooldownPeriod) {
                revert CooldownActive(_cooldownPeriod - elapsed);
            }
        }
    }
}
```

**措施说明**：

- 用户可设置操作冷却期
- 防止恶意高频操作
- 保护用户资产安全

#### 最大金额限制

```solidity
function setUserConfig(
    bool _enabled,
    uint256 _maxHedgeAmount,
    uint256 _cooldownPeriod,
    bool _autoExecute
) external {
    userConfigs[msg.sender] = UserConfig({
        // ...
        maxHedgeAmount: _maxHedgeAmount,
        // ...
    });
}
```

**措施说明**：

- 用户自定义最大对冲金额
- 防止单次操作造成过大损失
- 支持用户风险偏好设置

### 4.4 TEE 验证安全

#### 证明验证机制

```solidity
function executeWithTEEProof(
    address _user,
    uint8 _actionType,
    uint256 _amount,
    address _asset,
    bytes32 _inputHash,
    bytes32 _outputHash,
    bytes calldata _proof
) external nonReentrant {
    bool verified = TEEDecisionVerifier(teeVerifier).verifyDecision(
        _user,
        _actionType,
        _amount,
        _asset,
        _inputHash,
        _outputHash,
        _proof
    );

    if (!verified) {
        revert TEEVerificationFailed();
    }
    // ...
}
```

**措施说明**：

- 所有敏感操作需要 TEE 证明验证
- 证明包含输入输出哈希校验
- 支持远程验证和本地验证

## 五、性能优化

### 5.1 合约层面

| 优化项 | 实现方式 | 效果 |
|--------|----------|------|
| Storage 优化 | 结构体打包 | 减少存储 gas |
| View 函数 | 仅读取状态不消耗 gas | 降低调用成本 |
| 事件日志 | 使用事件代替存储 | 大幅降低存储成本 |
| 批量操作 | 合并多次调用 | 减少网络开销 |

### 5.2 应用层面

| 优化项 | 实现方式 | 效果 |
|--------|----------|------|
| 前端缓存 | React Query 缓存 | 减少重复请求 |
| 请求批量 | GraphQL/批量 RPC | 减少网络往返 |
| 前端渲染 | SSR/ISR | 提升首屏加载 |

### 5.3 计算层面

| 优化项 | 实现方式 | 效果 |
|--------|----------|------|
| AI 模型压缩 | 模型量化剪枝 | 降低推理延迟 |
| 缓存预测 | Redis/Memcached | 减少重复计算 |
| 异步处理 | 消息队列 | 提高吞吐量 |
